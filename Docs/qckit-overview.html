1. Luồng hoạt động tổng thể

Luồng được thực hiện tuần tự từ trái sang phải. Sau khi triage, nếu còn lỗi hoặc cần cập nhật selector, hệ thống quay lại giai đoạn chạy test và phân tích kết quả.

1

Khởi tạo

Chuẩn hóa Playwright, BasePage, session và danh sách URL.

qc_pre
2

Khám phá

Crawl DOM, component, hành động và user flow thực tế.

qc_detect_component
3

Mapping

Chuyển hiểu biết UI thành Locator, Page Object và system map.

qc_map_flow
4

Sinh Test Case

Sinh testcase, dữ liệu giả và tài liệu knowledge cho từng PBI.

qc_generate
5

Chạy Test

Sinh spec.ts, thực thi, trace, screenshot và cập nhật trạng thái.

qc_auto_test
6

Triage

Phân loại PASS, APP_BUG, INFRA và xác định severity.

qc_triage
Chu trình cải tiến: Fix lỗi hoặc cập nhật selector → chạy lại qc_auto_test → phân tích lại bằng qc_triage.

2. Chi tiết từng giai đoạn

Mỗi giai đoạn bên dưới thể hiện mục tiêu, input, output và vai trò kiểm soát của QC.

Giai đoạn 1

Cài đặt & Khởi tạo

Luồng: qc_pre gọi qc_setup, kiểm tra qc_basepage, sau đó crawl navigation để sinh aliases.

Input

  • .specify/tests/.env
  • .env.{ENV}
  • APP_URL
  • Chromium bundle

Output

  • playwright.config.ts
  • global-setup.ts
  • BasePage.ts
  • url-aliases.md
Vai trò QCXác nhận môi trường chạy được, tài khoản test hợp lệ, URL đầy đủ và đúng phạm vi cần kiểm thử.
Giai đoạn 2

Khám phá Ứng dụng

Luồng: qc_detect_component → qc_component_rule → qc_user_flow.

Input

  • url-aliases.md
  • component-rule.md
  • user.json
  • App đang chạy

Output

  • component_{alias}.md
  • component-rule.md
  • userflow-{group}.md
  • flow-index.md
Vai trò QCRà soát component, field, validation, toolbar action, happy path và các luồng phụ.
Giai đoạn 3

Mapping Page Object

Luồng: qc_map_flow → qc_url_page_map → qc_sub-system-map.

Input

  • component_{alias}.md
  • component-rule.md
  • userflow-{group}.md
  • url-aliases.md

Output

  • {Alias}Locator.ts
  • {Alias}Page.ts
  • locator-map.md
  • sub-system-map.json
Vai trò QCXác nhận locator ổn định, action method đúng nghiệp vụ và mapping đúng màn hình.
Giai đoạn 4

Sinh Test Case

Luồng: qc_generate <PBI_ID> đọc spec, plan và user flow để sinh toàn bộ bộ test.

Input

  • spec.md
  • plan.md
  • userflow-{group}.md
  • domain knowledge

Output

  • testcase.md
  • datafake.json
  • screen-summary.md
  • field-catalog.md
Vai trò QCReview độ phủ, nhóm test, priority, Auto/Manual, dữ liệu test và tiêu chí chấp nhận.
Giai đoạn 5

Chạy Test Tự động

Luồng: qc_auto_test <PBI_ID> sinh spec.ts, chạy theo thứ tự ưu tiên và ghi bằng chứng.

Input

  • testcase.md
  • Page Object / Locator
  • datafake.json
  • user.json

Output

  • {PBI_ID}.spec.ts
  • trace per TC
  • screenshot lỗi
  • HISTORY.md + summary
Vai trò QCĐánh giá precondition, độ tin cậy của assertion, data chaining, trace và trạng thái từng TC.
Giai đoạn 6

Phân loại & Triage

Luồng: qc_triage <PBI_ID> đọc run gần nhất, phân loại và xác định severity.

Input

  • HISTORY.md
  • run-summary.md
  • testcase.md
  • Trace / screenshot

Output

  • PASS
  • APP_BUG
  • INFRA
  • triage-{PBI_ID}-{DATE}.md
Vai trò QCƯu tiên loại trừ INFRA trước, sau đó xác nhận APP_BUG và đánh severity phù hợp.

4. Vai trò trung tâm của QC

QC kiểm soát đầu vào

Spec, plan, URL, session, dữ liệu test và phạm vi màn hình phải đủ và đúng trước khi automation chạy.

QC kiểm soát độ phủ

QC quyết định nhóm test, mức ưu tiên, TC nào Auto hay Manual và tiêu chí nào là đủ để chấp nhận PBI.

QC kiểm soát độ tin cậy

QC đánh giá assertion, trace, screenshot, session và data chaining để tránh false positive hoặc false negative.

QC quyết định phân loại lỗi

QCKit hỗ trợ tổng hợp bằng chứng, nhưng QC là người quyết định lỗi ứng dụng, lỗi hạ tầng và severity.
