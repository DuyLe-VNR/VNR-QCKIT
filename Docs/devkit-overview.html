Quy Trình VNR Auto-Pipeline

Pipeline tự động từ spec → report — tối đa 2 điểm dừng HITL: Duyệt Plan (luôn có) + Xác nhận Stack (chỉ khi phát hiện >1 UI stack). Mọi bước còn lại chạy tự động.

4. Chi Tiết Từng Bước

Bước 0 Kiểm tra & Khởi tạo CLI
Chức năng
  • Kiểm tra specs/<feature>/spec.md tồn tại (dừng ngay nếu thiếu)
  • Phát hiện cấu trúc repo từ docs/wiki/index.md (tag: architecture / recipe); dự phòng: quét src/ tìm dấu hiệu dự án (.sln, angular.json, pubspec.yaml)
  • Tạo branch feature/<feature> trong mỗi repo phát hiện được nếu chưa có
  • Kiểm tra ledger bền vững (TaskList) tìm các lần chạy bị gián đoạn → AskUserQuestion để tiếp tục từ phase phát hiện được
  • Tạo panel TaskCreate cho TẤT CẢ các phase (cùng lúc) với [$FEATURE] <phase.name>
Điều kiện bỏ qua
Nếu truyền --from=<phase-id>, bỏ qua hoàn toàn bước kiểm tra. Mọi phase trước đó được đánh dấu completed và orchestrator nhảy thẳng tới phase được chỉ định.
Bước 1a Plan (Lập kế hoạch) Skill: vnr-plan
Cách gọi
Skill tool: vnr-plan
Đọc nội bộ: agents/vnr-planner.md
Đầu ra
  • specs/<feature>/plan.md
  • specs/<feature>/data-model.md
  • specs/<feature>/contracts/
  • specs/<feature>/research.md
Tự động — Duyệt Plan (Bước 1b) chạy ngay sau đó.
Bước 1b ✦ MỚI Duyệt Plan Agent: vnr-plan-reviewer
Nội dung kiểm tra (18 mục)
  • P-01..P-05: Độ phủ Spec (AC, BR, VM, Data Dict, NEEDS CLARIFICATION)
  • P-06..P-09: API & Contracts (endpoint, permission key, wrapper)
  • P-10..P-14: Tuân thủ kiến trúc (Clean Arch, CQRS, các tầng)
  • P-15..P-18: Khả thi & Đầy đủ (thứ tự phase, nhất quán FK)
Logic kết luận
  • 🔴 Critical ≥ 1 → FAIL ⛔
  • 🟡 Warning ≥ 1, Critical = 0 → WARN ⚠️
  • Cả hai = 0 → PASS ✅
Luồng HITL (sau khi plan-reviewer hoàn tất):
1. Tùy điều kiện (tự bỏ qua nếu chỉ 1 stack): Nếu plan phân giải >1 UI stack → Xác nhận Stack qua AskUserQuestion; stack đã xác nhận được ghi lại vào mục ## Stack & Constraints của plan.
2. Luôn có: Duyệt chính → Con người quyết định: [approve → tiếp tục] / [modify → sửa plan.md thủ công rồi duyệt lại] / [reject → --from=plan]
Bước 1c Tasks (Chia task) Skill: vnr-tasks
Cách gọi
Skill tool: vnr-tasks
Đọc nội bộ: agents/vnr-task-breaker.md
Đầu ra
  • specs/<feature>/tasks.md
Chứa N task được tổ chức theo các nhóm song song.
Tự động — Tasks xong → Testcase Writer (Bước 2) chạy ngay. Không cần xác nhận từ người dùng.
Bước 2 Testcase Writer Agent: vnr-testcase-writer
Chức năng
  • Đọc wiki, spec, plan, tasks, contracts, chi tiết UI
  • Đầu ra: testcases.md
  • Phân loại: API, UI, Authorization, Integration
  • Độ ưu tiên: P0 (Nghiêm trọng) / P1 (Cao) / P2 (Trung bình) / P3 (Thấp)
Đầu ra
  • specs/<feature>/testcases.md
Tự động — Testcases xong → Triển khai (Bước 3) chạy ngay. Không cần xác nhận từ người dùng.
Bước 3 Triển khai (Implement) Skill: vnr-implement
Cách gọi
Skill tool: vnr-implement
Tự động nhận diện scope từ tiền tố đường dẫn trong tasks.md — điều phối tuần tự:
src/backend/ → vnr-backend-developer
src/frontend/ → vnr-frontend-developer
src/app-mobile/ → vnr-mobile-developer
Chức năng
  • Nạp tasks.md + plan.md + contracts + data-model
  • Thực thi task theo từng phase, đánh dấu [x] khi xong
  • Dừng nếu một task không song song bị lỗi
  • Build phải pass (0 lỗi)
Tự động — Build thành công → Bước 4 chạy ngay. Tự dừng nếu build lỗi — người dùng sửa rồi --from=implement.
Bước 4 ✦ MỚI Review Code Agent: vnr-code-reviewer
Nội dung kiểm tra
  • Đầy đủ AC/BR — mọi AC trong spec.md đều có code
  • Đầy đủ Task — mọi task trong tasks.md đều được hiện thực
  • Chất lượng code — đặt tên, DRY, code chết, xử lý lỗi, an toàn null
  • Đúng logic — nhánh điều kiện, off-by-one, async/await
  • Hardcode / Magic Number — không có giá trị cứng đáng lẽ phải là config
Logic kết luận
  • 🔴 Critical ≥ 1 hoặc thiếu AC → FAIL ⛔
  • 🟡 Warning ≥ 1, Critical = 0 → WARN ⚠️
  • Cả hai = 0 → PASS ✅
Đọc: spec.md, tasks.md, quy ước wiki, constitution
Tự dừng nếu FAIL — sửa lỗi logic/AC rồi --from=code-review. PASS/WARN → Bước 5+6 chạy ngay.
Bước 5 + 6 Review Kiến trúc + Review Bảo mật (SONG SONG) Agent Song song
Review Kiến trúc (Bước 5)
  • Agent: vnr-arch-reviewer
  • Đọc: chuẩn kiến trúc, contracts, constitution
  • Chạy: 14 mục BE + 10 mục FE
  • Kết luận: PASS / WARN / FAIL
Review Bảo mật (Bước 6)
  • Agent: vnr-sec-reviewer
  • Đọc: chuẩn permission, security-hooks, constitution
  • Chạy: checklist OWASP (14 BE + 7 FE)
  • Kết luận: PASS / WARN / FAIL
Tự động — PASS/WARN → Bước 7 chạy ngay. Tự dừng nếu FAIL — người dùng sửa rồi --from=arch-sec-review.
Bước 7 E2E Stubs (Khung mẫu) Agent (inline) LUÔN PASS
Chức năng
  • Kiểm tra src/frontend/e2e/<feature>.e2e.spec.ts có tồn tại không
  • Nếu chưa: tạo stub tối thiểu với test.todo() cho các testcase P0+P1
  • KHÔNG chạy bất kỳ test nào
  • Luôn pass — trạng thái: ⏭ STUB
Điều phối
type: agent với inline_prompt trong workflow.yml — không có file agent riêng.
Đọc docs/wiki/index.md để tìm đường dẫn recipe E2E; dự phòng dùng src/frontend/e2e/{feature}.e2e.spec.ts.
Stub tham chiếu testcases.md (P0+P1). Đội automation hiện thực phần thân riêng.
Không có điểm dừng — tự động chuyển sang Bước 8. Luôn ⏭ STUB.
Bước 8 Báo cáo (Report) Agent: vnr-tech-writer
Cách gọi
Agent tool (general-purpose)
Đọc: agents/vnr-tech-writer.md
Đầu ra
  • specs/<feature>/result/final-report.md
  • specs/<feature>/result/user-guide.md
Bao gồm: bảng tổng hợp (Kiến trúc, Bảo mật, E2E Stubs), phát hiện, file thay đổi, checklist sign-off, hướng dẫn sử dụng bằng tiếng Việt.

6. Điều Kiện Cổng (theo Constitution)

Chuyển tiếp Điều kiện cổng Khi thất bại
Bước 1a → 1b Tự động — Duyệt Plan chạy ngay —
Bước 1b → 1c plan.md được người dùng duyệt (sau khi xem phát hiện của plan-reviewer) Từ chối → sửa plan → --from=plan
Bước 1c → 2 Tự động — Tasks xong → Testcase Writer chạy ngay —
Bước 2 → 3 Tự động — Testcases xong → Triển khai chạy ngay —
Bước 3 → 4 Tự động nếu build thành công (0 lỗi) — cả BE và FE Tự dừng, sửa rồi --from=implement
Bước 4 → 5+6 Tự động nếu Review Code PASS/WARN Tự dừng nếu FAIL, sửa rồi --from=code-review
Bước 5+6 → 7 Tự động nếu Kiến trúc PASS/WARN + Bảo mật PASS/WARN Tự dừng nếu FAIL, sửa rồi --from=arch-sec-review
Bước 7 → 8 Tự động — stub luôn pass —
Bước 8 → Hoàn tất Tự động — pipeline hoàn tất, artifacts sẵn sàng cho PR —

7. Khôi Phục & Tiếp Tục

# Tiếp tục từ một phase cụ thể (tái sử dụng artifacts hiện có)
/vnr-auto-pipeline <feature> --from=plan ← chạy lại Plan (planner)
/vnr-auto-pipeline <feature> --from=plan-review ← chạy lại riêng Duyệt Plan
/vnr-auto-pipeline <feature> --from=tasks ← từ Tasks (plan đã duyệt)
/vnr-auto-pipeline <feature> --from=testcases ← từ Testcase Writer
/vnr-auto-pipeline <feature> --from=implement ← từ Triển khai (đã sửa build)
/vnr-auto-pipeline <feature> --from=code-review ← từ Review Code (đã sửa logic/AC) ✦ MỚI
/vnr-auto-pipeline <feature> --from=arch-sec-review ← từ Review Kiến trúc+Bảo mật (đã sửa arch/sec)
/vnr-auto-pipeline <feature> --from=e2e-stubs ← từ bước kiểm tra E2E Stubs
/vnr-auto-pipeline <feature> --from=report ← chỉ Báo cáo

# Khi dùng --from=<phase-id>: bỏ qua kiểm tra các bước trước,
# tái sử dụng artifacts hiện có (plan.md, tasks.md, v.v.)
